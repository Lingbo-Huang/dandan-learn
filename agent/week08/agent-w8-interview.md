## 一、基础架构类

### Q1：请描述 ReAct Agent 的工作原理

**标准答案（STAR 结构）：**

ReAct（Reasoning + Acting）是当前最主流的 Agent 架构，核心思路是让 LLM 交替进行推理和行动：

```
用户问题
   ↓
Thought（推理）：我需要先搜索最新数据...
Action（行动）：search_tool("xxx")
Observation（观察）：搜索结果是...
Thought（推理）：根据结果，我需要进一步分析...
Action（行动）：calculate(...)
Observation（观察）：...
Final Answer：综合以上信息，答案是...
```

**关键要点：**
- LLM 负责规划和决策，工具负责执行
- 每一步的 Observation 作为新的上下文输入
- 通过 `AgentExecutor` 的循环机制实现多步推理

**追问：ReAct 有什么局限性？**
1. 线性推理，无法并行执行工具
2. 长链路下 Context 增大，Token 消耗高
3. 容易陷入循环（需要 max_iterations 限制）

### Q2：多 Agent 系统相比单 Agent 的优缺点？

**优点：**
- **专业化分工**：每个 Agent 专注一个领域，质量更高
- **并行处理**：多 Agent 可同时工作，效率更高
- **容错**：单个 Agent 失败不影响整体（有 orchestrator 兜底）
- **可扩展**：新增能力只需添加新 Agent

**缺点：**
- **通信开销**：Agent 间协调有额外的 LLM 调用成本
- **调试困难**：分布式系统的 Bug 更难排查
- **一致性挑战**：多 Agent 对同一问题可能得出矛盾结论
- **复杂性**：设计合理的协作协议需要大量工程投入

**回答技巧**：举一个具体场景，说明什么情况下用多 Agent（大规模并行任务、专业度要求高），什么情况下用单 Agent（交互简单、成本敏感）。

### Q3：如何设计一个高可用的 Agent 系统？

**分层回答：**

```
1. 应用层：重试机制 + 熔断器 + 降级策略
   - 指数退避重试（3次，最大30秒）
   - 熔断器（5次失败触发，60秒恢复）
   - 模型降级（gpt-4o → gpt-4o-mini → 固定模板）

2. 基础设施层：无状态部署 + 水平扩展
   - 会话状态存 Redis，Pod 无状态
   - K8s HPA 自动扩缩容（CPU 70% 触发）
   - 多 AZ 部署，避免单点故障

3. 数据层：缓存 + 幂等
   - Redis 缓存热点 LLM 响应
   - 所有操作设计幂等（重试安全）

4. 可观测层：日志 + 指标 + 追踪
   - 结构化日志（trace_id 贯穿全链路）
   - 关键指标告警（错误率>10%、延迟>5s）
```

### Q4：如何评估 Agent 的效果？

**完整的评估体系：**

| 维度 | 指标 | 评估方法 |
|------|------|---------|
| 正确性 | 任务完成率、答案准确率 | LLM 评估 + 人工标注 |

### Q5：如何控制 Agent 的 Token 消耗和成本？

**系统性回答：**

```python
# 1. 模型分级：简单任务用小模型
def select_model(task_complexity: str) -> str:
    return {
        "simple": "gpt-4o-mini",   # 节省 95% 成本
        "standard": "gpt-4o-mini",
        "complex": "gpt-4o"
    }[task_complexity]

# 2. 响应缓存：相同问题不重复调用
# 命中率 30-80% 可节省大量成本

# 3. Prompt 压缩：减少输入 Token
# 将长上下文压缩到 1/3-1/5

# 4. 输出控制：限制最大输出长度
llm = ChatOpenAI(max_tokens=500)  # 非必要不超长

# 5. 并行调用：减少等待时间（不减少 token，但提升效率）
```

**量化表达**：告诉面试官"通过模型分级 + 缓存，我们把平均调用成本从 $0.05 降到了 $0.008，降低了 84%"。

### Q6：如何防止提示注入攻击？

**防御纵深：**

1. **输入过滤**：规则 + LLM 双层检测
2. **权限最小化**：工具按用户权限白名单开放
3. **沙箱隔离**：代码执行在独立容器，无网络
4. **输出审计**：发送前过滤敏感信息
5. **异常监控**：检测到注入模式立即告警

## 三、系统设计类

### Q7：设计一个企业内部知识库问答系统

**这是最常见的大题，标准答案框架：**

```
需求澄清（先问）：
- 知识库规模（百 GB 级？千 GB 级？）
- 并发用户数（QPS？）
- 对准确性要求（幻觉容忍度？）
- 是否需要多轮对话？

系统设计：

1. 知识入库：
   文档 → 分块（Chunk: 512 tokens，重叠 64 tokens）
        → 向量化（text-embedding-3-small）
        → 存向量DB（Qdrant/Chroma）
        → 存原文（对象存储）

2. 查询流程：
   问题 → 改写（HyDE）→ 向量检索（Top-K）
        → 重排序（Cross-Encoder）→ 上下文组装
        → LLM 生成回答 → 溯源引用

3. 可靠性：
   - 答案无法回答时明确说 "我不知道"（Confidence Score）
   - 所有回答附带引用来源
   - 定期更新知识库（增量索引）

4. 评估：
   - Ragas 框架：Faithfulness / Answer Relevancy / Context Recall
```

### Q8：你遇到过最难的 Agent Bug 是什么？如何解决的？

**STAR 结构回答：**

**S（情景）**：我们的研究 Agent 会随机给出错误的引用（幻觉），但在测试环境复现不了。

**T（任务）**：找到根本原因，消除幻觉，不降低整体性能。

**A（行动）**：
1. 加入 LangSmith 全链路追踪，记录每次 Token 输入输出
2. 发现问题在于 Context 长度超过 4k 时，模型开始"凑"引用
3. 添加了 Faithfulness 评估器，实时检测回答是否基于 Context
4. 改为只输出有检索依据的内容，并附来源链接

**R（结果）**：幻觉率从 23% 降到 4%，用户满意度提升 18%。
